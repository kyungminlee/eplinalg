# Pipeline refactor — symmetric patch + migrate (2026-05-09)

## Context

The current `migrator stage` pipeline migrates one half of each co-family
pair (D/Z by default, S/C if `prefer_source:` flips it) and discards the
other. `source_overrides` patch the half that becomes canonical, but
never the sibling. Three consequences:

1. **kind4/kind8 baseline still buggy.** That path links unmodified
   `external/`, so `dorbdb3`/`zunbdb3` and 6+ other already-fixed bugs
   still SIGSEGV / return wrong numerics in the baseline column.
   `tests/lapack/TODO.md` parks these.
2. **Divergence reports mix two signals.** "Real upstream design
   asymmetry" and "we patched D, forgot to patch S" both surface as
   diffs, so audit time goes into hand-classifying.
3. **Full-file overrides hide upstream evolution.** When Netlib bumps
   `external/`, an override that no longer represents a fix-on-top of
   upstream is indistinguishable from a fix that's still needed. No
   signal that upstream may have already addressed the bug.

The refactor reshapes the pipeline into six explicit stages so both
halves are patched symmetrically, patches are stored as unified diffs,
and divergence becomes pure signal.

## Target pipeline

```
(1) prepare    : copy external/<lib>/ → build/staged-sources/<lib>/   (per recipe, idempotent)
(2) patch      : apply recipes/<lib>/patches/*.patch to staged-sources (unified diffs vs upstream)
(3) migrate    : run BOTH halves of every co-family pair through the migrator (per target)
(4) diverge    : compare migrated halves; emit divergence report
(5) pick       : choose canonical per pair via prefer_source rank;
                 converged pairs trivial, divergent pairs error unless whitelisted
(6) build      : as today
```

Stages 1+2 run once per upstream tree (target-independent). Stages 3-6
run per target.

## Patch file convention

```
recipes/<lib>/patches/<BUG_ID>__<short-slug>.patch
```

- `<BUG_ID>` matches `doc/UPSTREAM_BUGS.md` (`L01`, `S04`, `I07`, `X02`, …).
- Unified diff against `external/<lib>/<vendor-version>/` (multi-file
  allowed: one patch can touch all four halves of a co-family).
- Applied with `git apply --check` before `git apply`. **A reject is
  a hard error**, not a warning — the prepare stage stops, the user
  must reconcile (likely upstream moved; re-audit whether the bug
  still exists).
- Recipes list patch filenames in `patches:` to give explicit ordering;
  missing files in the directory are an error.

This format gives three signals upstream-evolution-wise that today's
full-file overrides do not:

- **Clean apply, hunk still present** → bug unfixed upstream, our patch
  still needed.
- **Reject** → upstream touched the area; re-audit required.
- **Empty patch (no-op)** → upstream fixed it; the patch file (and the
  `UPSTREAM_BUGS.md` entry) can be retired.

## Recipe schema diff

```yaml
# Before
source_overrides:
  dorbdb3.f: source_overrides/dorbdb3.f
  sorbdb3.f: source_overrides/sorbdb3.f
  zorbdb3.f: source_overrides/zorbdb3.f
  corbdb3.f: source_overrides/corbdb3.f
prefer_source: [PZUNGQL, ...]

# After
patches:
  - L01__orbdb3-rot-stride.patch       # touches d/s/z/c orbdb3.f symmetrically
  - L02__dgejsv-lwork-overflow.patch
  - ...
prefer_source: [PZUNGQL, ...]          # unchanged
```

`prefer_source:` stays a flat uppercase-stem set. No per-target
conditioning.

## Execution phases

### Phase A — patch infra, no behavior change

1. Create this document (`doc/refactor-20260509.md`).
2. Define `recipes/<lib>/patches/` directory convention.
3. Plumb `RecipeConfig.patches` validation: each entry must exist in
   `recipes/<lib>/patches/`. Loader-only; no apply yet.
4. Add `migrator prepare <recipe> [--rebuild]` CLI command. Copies
   `external/<lib>/` → `build/staged-sources/<lib>/`, then `git apply`
   each listed patch in order. Idempotent: if
   `build/staged-sources/<lib>/.prepared.stamp` is newer than the
   upstream tree and all listed patches, skip; `--rebuild` forces.
5. Add `.gitignore` rule for `build/staged-sources/`.
6. **Verify A:** `migrator prepare recipes/lapack.yaml` with empty
   `patches:` produces a tree byte-identical to
   `external/lapack-3.12.1/` (use `diff -r`).

### Phase B — migration reads from staged-sources

1. Resolve `config.source_dir` for migration to
   `build/staged-sources/<lib>/SRC` (or whatever upstream layout
   dictates) instead of `external/<lib>/.../SRC`. One spot in the
   `RecipeConfig.source_dir` derivation; see `config.py:35-183`.
2. `migrator run` chains `prepare → migrate → converge → verify → build`.
3. `migrator stage`'s kind4/kind8 baseline path stages from
   `build/staged-sources/<lib>/` rather than raw `external/`.
4. **Verify B:** With patches still empty, every existing test
   continues to pass. `/tmp/stage-q` outputs are byte-for-byte
   unchanged.

### Phase C — convert source_overrides to patches (one library at a time)

Order: blas (4) → blacs (2) → pblas (2) → scalapack (29) → lapack (64).
Total 101 conversions.

For each override file:

1. Generate unified diff:
   `diff -u external/<lib>/.../<orig> recipes/<lib>/source_overrides/<orig>`
   → `recipes/<lib>/patches/<BUG_ID>__<slug>.patch`.
2. Where multiple overrides fix the same bug across halves (e.g.,
   `?orbdb3` × 4), consolidate into one patch with four file hunks.
3. Add patch to recipe's `patches:` list; remove from `source_overrides:`.
4. Run `migrator run --target kind16` for that library; compare output
   tree against pre-conversion baseline (saved snapshot). Must be
   byte-identical.
5. After all overrides for a library converted: delete the recipe's
   `source_overrides:` block, delete `recipes/<lib>/source_overrides/`,
   commit.

After all 11 recipes converted: delete `RecipeConfig.source_overrides`,
delete the resolution code at `pipeline.py:704-708`.

**Verify C (per library):** byte-identity of migrated output before vs
after conversion.

### Phase D — symmetric-patch enforcement, divergence becomes hard error

1. Add a CI check: every patch under `recipes/<lib>/patches/` that
   touches a precision-prefixed file must touch all four halves (or
   all halves the recipe declares for that family). Detected by
   parsing patch filenames within the diff; allowed exceptions listed
   in a per-recipe `asymmetric_patches:` field.
2. Convergence runs after `pick` and emits a hard error on any
   divergent pair not listed in `expected_divergences:` (new recipe
   field; superset of today's "leave as known-divergent" cohort: BLAS
   sdot line-swap, srotmg constants, MINRGP family, MUMPS S/D
   source-level drift, etc.).
3. **Verify D:** Convergence count drops; remaining divergences are
   exactly the upstream-design cohort enumerated in
   `doc/convergence-20260507.md` "expected" notes.

### Phase E — close kind4/kind8 baseline gaps

1. Run baseline tests against the staged-sources tree.
2. Confirm L01 / L07 / L11 etc. clear in baseline (the bugs that today
   only fix in the migrated archive).
3. Update `tests/lapack/TODO.md`: remove parked items now unblocked.
4. Document in `tests/KIND48_BASELINE_STATUS.md`.

## Verification end-to-end

After Phase E, the canonical regression should be:

```bash
rm -rf build/staged-sources /tmp/stage-q
uv run python -m migrator run recipes/lapack.yaml /tmp/stage-q --target kind16
uv run python -m migrator run recipes/scalapack.yaml /tmp/stage-q --target kind16

for lib in blas blacs lapack pblas scalapack scalapack_c mumps ...; do

        /tmp/stage-q/$lib/src --target kind16
```

whitelist, OR is one of the three deferred migrator-side gaps.

## Risks and mitigations

| Risk                                                            | Mitigation                                                                                            |
|-----------------------------------------------------------------|-------------------------------------------------------------------------------------------------------|
| `git apply` is finicky about line endings / whitespace          | Pin to `git apply --whitespace=nowarn`; CI runs prepare on clean checkout                             |
| 101 overrides → 101 conversions is tedious                      | Phase C scripted: `diff -u` generation, recipe edit, regression compare. One library per commit.     |
| Some overrides are full rewrites, not minimal patches           | Those become large patches — that's fine, they're still diffs and still surface upstream churn       |
| Divergence-as-hard-error breaks dev flow if list is incomplete  | Phase D ships `expected_divergences:` pre-populated from current `doc/convergence-20260507.md` cohort |
| `extra_c_dirs` / `extra_fortran_dirs` may point into `external/`| Audit all 11 recipes for absolute `external/` paths; redirect to `staged-sources/` during Phase B    |

## Open questions for the implementer

1. **Patch base path.** Vendor version pinned per recipe
   (`external/lapack-3.12.1/`) or symlink (`external/lapack/`)? Today
   recipes hard-code the version directory. A vendor bump means
   re-pathing; new mechanism should make that one-line.
2. **`prepare` cache key.** Stamp file or content hash? Stamp is
   simpler; hash detects out-of-band edits to staged-sources.
3. **Hand-rewritten C files** like multifloats `BI_qMPI_sum`. Today
   they live in `recipes/<lib>/overrides/` (post-migration target-gated
   copy), not `source_overrides/`. They're orthogonal to this refactor
   and stay where they are.
