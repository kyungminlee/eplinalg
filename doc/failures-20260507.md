# Aggregator / test-driver bugs surfaced during RESULT.md regen — 2026-05-07

The 2026-05-07 e2e run (1 151 / 1 151 PASS on every target) exposed a
set of *latent* bugs in the precision-report writer/aggregator path:
the affected tests pass under `ctest`, but their JSON sidecars are
either malformed or missing required fields. None of these affect
ctest pass/fail; all of them affect the offline aggregation that
produces `tests/REPORT.md` / `tests/RESULT.md`.

`scripts/result_report.py` already tolerates them (warns + skips),
so RESULT.md is consistent. `scripts/precision_report.py` does not —
it crashes on the missing-`digits` group (B below). The earlier
`doc/failures-20260506.md` cross-cutting note (#4) attributed the
parse failures to literal `Infinity` / `-Inf` JSON tokens; that
diagnosis was wrong. The `Infinity` cases were the un-renamed
`ref_quad_*` failures (`dorcsd`, `zvvdotc/u`) that landed actual fixes
on 2026-05-06; what's left here is a different class — truncated
files plus a missing field.

## A. Truncated JSON reports — `prec_report` writer (LAPACK suite)

### A.1. Report ends after `"cases": [` — no cases, no closing tokens

Affected (all three targets, identical pattern):

| routine    | bytes written        | aggregator error                                    |
|------------|----------------------|-----------------------------------------------------|
| `dorbdb5`  | header + `"cases": [`| `Expecting value: line 5 column 1 (char 62)`        |
| `dorbdb6`  | header + `"cases": [`| `Expecting value: line 5 column 1 (char 62)`        |
| `zunbdb5`  | header + `"cases": [`| `Expecting value: line 5 column 1 (char 62)`        |
| `zunbdb6`  | header + `"cases": [`| `Expecting value: line 5 column 1 (char 62)`        |

Sample (`dorbdb5.kind10.json`, full file content):

```json
{
  "routine": "dorbdb5",
  "target":  "kind10",
  "cases": [
```

The driver (`tests/lapack/factorization/test_dorbdb{5,6}.f90`,
`test_zunbdb{5,6}.f90`) calls `report_init`, runs its case loop, and
returns rc=0 — ctest #s 366/367 (`d*`) and 555/556 (`z*`) pass. Yet
neither the body cases nor the closing `]` / `}` from
`report_finalize` make it to disk. `prec_report.report_finalize`
writes `'  ]'` then `'  }'` then `close(unit_save)`; one of those
steps is being skipped in this driver shape. Likely candidates: an
`error stop` from the LAPACK call path that the test treats as a
`PASS` outcome (suppressing finalize), or an iostat path inside
`report_case` that bails before any case is written.

### A.2. Report ends mid-case — first case written, no closing tokens

Affected (all three targets):

| routine        | aggregator error                                         |
|----------------|----------------------------------------------------------|
| `dgetsqrhrt`   | `Expecting ',' delimiter: line 12 column 1 (char 236)`   |
| `zgetsqrhrt`   | `Expecting ',' delimiter: line 12 column 1 (char 236)`   |

Sample (`dgetsqrhrt.kind10.json`, full file content):

```json
{
  "routine": "dgetsqrhrt",
  "target":  "kind10",
  "cases": [
    {
      "case":        "m=16,n=4",
      "max_rel_err": 7.564280E-020,
      "tolerance":   4.440892E-016,
      "digits":      19.12,
      "passed":      true
    }
```

The driver (`test_dgetsqrhrt.f90`, `test_zgetsqrhrt.f90`) iterates
two `m,n` shapes; only the first case object is on disk and the
trailing `]\n}` is missing. Same root-cause family as A.1.

### A.3. Probable mechanism

Both A.1 and A.2 are consistent with `report_finalize` not running:
either the test program `error stop`s before the call, or — more
likely given that ctest reports PASS — a Fortran I/O sequencing
detail in the writer is dropping buffered output. The fact that A.2
has one case but A.1 has none suggests A.1's drivers aren't iterating
their case loop at all (perhaps an empty `ms(*)`, or a `deallocate`
ordering bug aborting before the first `report_case`).

Either way the fix is in `tests/lapack/common/prec_report.f90` plus
the four/two affected drivers; a defensive `flush` after each
`write` would already make A.2 inspectable, and a `report_finalize`
called from a Fortran end-program hook would protect against A.1.

## B. MUMPS C-bridge JSONs lack the `digits` field

Affected:

* `test_dmumps_c_basic.{kind10,kind16,multifloats}.json`
* `test_dmumps_c_sym.{kind10,kind16,multifloats}.json`
* `test_zmumps_c_basic.{kind10,kind16,multifloats}.json`

Sample (`test_dmumps_c_basic.kind10.json`):

```json
{
  "routine": "test_dmumps_c_basic",
  "target":  "kind10",
  "cases": [
    {
      "case":        "n=4",
      "max_rel_err": 1.084202e-19,
      "tolerance":   1.110223e-16,
      "passed":      true
    }
  ]
}
```

vs. the Fortran writer's per-case schema (`tests/lapack/common/
prec_report.f90`):

```json
{
  "case":        "...",
  "max_rel_err": ...,
  "tolerance":   ...,
  "digits":      ...,    ← missing in C-bridge
  "passed":      ...
}
```

The C-bridge tests under `tests/mumps/c/test_*mumps_c_*.c` emit
their own JSON without going through `prec_report`. The schema drift
crashes `scripts/precision_report.py`:

```
File "scripts/precision_report.py", line 108, in render_summary
    cells.append(fmt_digits(worst_case['digits']) + mark)
                            ~~~~~~~~~~^^^^^^^^^^
KeyError: 'digits'
```

`scripts/result_report.py` survives because `result_report.py:108`
calls `fmt_digits(case.get('digits'))`, treating absent as `None` →
`'—'`; it filters MUMPS out of its per-library tables anyway.

## C. Aggregator hardening — `scripts/precision_report.py`

Independent of the per-test fixes, two cheap robustness changes
would let the aggregator emit a complete summary regardless of test
output quality:

1. `render_summary` (`precision_report.py:108`) and `render_detail`
   (`:138`) should both use `case.get('digits')` and pass through
   `fmt_digits` (which already handles `None` → `'—'`).
2. `load_reports` already warns on `JSONDecodeError`; consider
   also verifying that `cases` is a list and skipping (with a
   warning) when it's missing, so a future writer regression that
   omits the wrapper doesn't crash later passes.

Neither change alters the published numbers — they widen what the
aggregator can ingest without crashing.

## Triage order

1. Fix B by extending the MUMPS C-bridge writer (`tests/mumps/c/
   *.c`) to emit `digits = -log10(max_rel_err)` (with `99.0` for
   `max_rel_err == 0`), matching the Fortran `prec_report` schema.
   Smallest-blast-radius change; restores `precision_report.py`.
2. Land C as a defense-in-depth follow-on so future schema drift
   degrades to `'—'` cells instead of `KeyError`.
3. Reproduce A by running `test_dorbdb5` standalone with `strace
   -e write` (or a compiler-side `flush` in `report_case`) to
   confirm whether `report_finalize` runs. If yes, the fix is in
   the writer; if no, the bug is in the driver. Either way the
   fix likely sits in `tests/lapack/common/prec_report.f90` plus
   the six affected drivers, and is invisible to ctest.

## Out of scope

Everything here is bookkeeping — none of these issues alter the
1 151 / 1 151 ctest outcome, the per-routine digits-of-agreement
already present in `tests/RESULT.md`, or the migrator's output. They
are aggregator/observability bugs.
